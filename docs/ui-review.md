# UI-аудит SmartOffers

## Что уже хорошо

- Приложение быстро ведёт пользователя к главному сценарию: определить район, найти ближайшие магазины и открыть каталоги.
- Карточки магазинов и офферов уже используют единый визуальный язык: скругления, светлые поверхности, акцентный зелёный цвет.
- Экран каталога разделяет товары по страницам буклета, что полезно, если пользователь сверяет приложение с бумажным/онлайн-флаером.

## Улучшения UI, которые стоит сделать следующими

1. **Сделать домашний экран более “shopping-first”.** Сейчас hero говорит про MVP и локацию. Лучше заменить это на ценность для пользователя: “Сравните актуальные акции рядом с вами”, добавить быстрые чипы популярных сетей и последние просмотренные каталоги.
2. **Унифицировать язык интерфейса.** В приложении смешаны русский, английский и испанский контент Tiendeo. Для MVP лучше выбрать русский UI, а названия магазинов/каталогов оставлять как есть.
3. **Добавить состояния пустых данных с действиями.** Например: “Каталоги не найдены — попробуйте увеличить радиус / выбрать другой магазин / открыть сайт Tiendeo”.
4. **Показывать превью и количество товаров на карточке каталога.** Пользователь должен понимать, что откроется, ещё до перехода в каталог.
5. **По умолчанию открывать весь каталог, а не первую страницу.** Первая страница часто содержит только часть товаров, поэтому пользователь воспринимает это как потерю данных. Фильтр по страницам должен быть дополнительным режимом.
6. **Добавить поиск/фильтры внутри каталога.** Минимально: поиск по названию товара, фильтр “с ценой”, сортировка по цене и чипы страниц.
7. **Сделать цены визуальным якорем.** В карточках товаров цена должна быть заметнее названия страницы, а отсутствие цены лучше показывать нейтральным бейджем, а не основным текстом.
8. **Показать источник и ограничения данных.** Для каталога стоит явно отображать “Источник: Tiendeo” и предупреждение, если Tiendeo вернул только часть товаров.

## Почему не было превью каталогов

Фронтенд пытался взять картинку из `item.products?.[0]?.image`, но эндпоинт списка каталогов возвращал только `id`, `title` и `url`. Из-за этого `previewImage` всегда был пустым, и карточка показывала плейсхолдер. Исправление: backend теперь обогащает каждый каталог `productCount` и `previewImage`, а экран магазина использует эти поля.

## Почему при открытии каталога казалось, что есть только 50 товаров

Экран каталога сразу выбирал первую страницу и запрашивал `/products?page=1`. Поэтому пользователь видел только товары одной страницы буклета. Исправление: экран теперь сначала загружает `/catalog/:id` целиком, показывает режим “Все” по умолчанию и оставляет страницы как фильтр.

## Что делать с Tiendeo, если всё равно возвращается только часть товаров

Tiendeo на публичной странице бренда показывает агрегированные счётчики офферов и каталогов. Например, для Lidl на 30 мая 2026 Tiendeo показывает 1096 офферов и 7 каталогов. Это больше, чем может быть доступно из первого HTML-снимка каталога. Если после исправления `/catalog/:id` всё ещё возвращает ровно 50 товаров, значит Tiendeo отдаёт только первый ленивый batch в `__NEXT_DATA__`, а остальные товары подгружаются внутренним API/клиентским запросом.

План решения:

1. Открыть каталог Tiendeo в браузере с DevTools → Network.
2. Отфильтровать `Fetch/XHR` по словам `crop`, `product`, `offer`, `flyer`, `catalog`.
3. Пролистать каталог до конца и найти запросы с параметрами пагинации (`page`, `offset`, `limit`, `cursor`, `from`).
4. Повторить эти запросы в backend-сервисе, пока API возвращает непустую страницу.
5. Объединять результаты с текущим парсером `__NEXT_DATA__`, дедуплицировать по названию, цене, картинке и странице.
6. Если Tiendeo требует cookies/геолокацию/одноразовые токены, хранить это только на backend и не раскрывать во фронтенде.

Важно: текущий код не должен пытаться “угадывать” приватные endpoints без наблюдения реального Network-запроса, потому что Tiendeo может менять имена route handlers и параметры между деплоями.
